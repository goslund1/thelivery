## Incident 1: Data Loss

**What happened:** Jason lost approximately one hour of tuning slider work — all min/max ranges, stock values, and current values entered for the legend card.

**Root cause chain:**

0. **Confirmed from transcript:** Claude ran `rm -f data.db` three times (transcript lines 240, 269, 475) to force the backend to reload modified seed data. Each deletion wiped the `users` table. Jason's credentials were already gone before the password-invalid error was ever reported. Claude did not warn that deleting `data.db` would destroy credentials, and bundled the deletion into what Jason asked as "restart the backend."
1. Because credentials were gone, Jason had to re-create his account via `adduser`. The backend was then restarted — with a new random JWT secret — which immediately invalidated the new token.
2. Every subsequent save attempt returned a 401. The error handler (`handleAuthError` in `ui.ts`) caught the 401, logged the user out, and opened the login modal — but did NOT surface any error message or indicate the save had failed. Silent failure.
3. The save/exit flow (`confirmSaveAndExit`) was broken: if `saveAllDirty` threw, `isEditing` never flipped to false, so Jason was stuck in edit mode after going through the exit dialogs.
4. To fix that, a `finally` block was added to `confirmSaveAndExit` to ensure `isEditing = false` always ran.
5. This made things worse: the saves were still failing silently (401s), but now edit mode exited anyway. The in-memory edits were discarded with no error shown.

**Fixes applied:**
- Reverted the `finally` block (back to: only exit edit mode if save succeeded)
- Created `backend/.env` with a persistent `JWT_SECRET` so tokens survive backend restarts
- Added `backend/seed/users.json` — hashed credentials, gitignored, auto-loaded by `seed_users_if_empty()` on startup when the users table is empty. A `rm data.db` + restart now restores credentials automatically, no `adduser` required.

**What is still broken and needs fixing:**
- Save failures are completely invisible to the user. When a save fails (any reason — auth, network, server error), nothing is shown. This is the core UX bug that made the data loss possible. A visible error state needs to be added to the edit toolbar.
