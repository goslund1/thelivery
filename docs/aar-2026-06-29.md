# After Action Report — Session 2026-06-29

## What we were working on

`UpgradesPicker.vue` and `RecipeSection.vue` — responsive 3→2→1 column layout for the upgrades list.

---

## The problem

Six upgrade categories need to display in 3 columns at wide widths:

```
Col 1                  Col 2                      Col 3
─────────────────────  ─────────────────────────  ─────────────────────
Body Kits + Engine     Drivetrain                 Tires and Wheels
                       Platform and Handling
                       Aero and Appearance
```

At 2 columns:

```
Col 1                  Col 2
─────────────────────  ─────────────────────────
Body Kits + Engine     Platform and Handling
Drivetrain             Aero and Appearance
                       Tires and Wheels
```

---

## What we tried that failed

### Approach 1: CSS grid with bundled kit-cols

Grouped categories into 3 kit-col DOM nodes (Body Kits+Engine / Drivetrain+Platform+Aero / Tires). Used CSS `order` to rearrange at 2-col. Problem: `order` moves the entire bundle — you can't split Drivetrain from Platform+Aero. Drivetrain is not a separate DOM item so it can't move independently.

### Approach 2: CSS grid with 4 kit-cols and explicit row placement

Split to 4 DOM items; used `grid-column`/`grid-row` to place them. Fatal flaw: **CSS grid shares row heights across all columns in the same row.** Engine has 14 items (~400px), Drivetrain has 4 items (~130px). Row 1 = max of both = Engine height. Drivetrain sits at the top of a 400px row with 270px of dead space below it before Platform+Aero starts. Looked terrible at every breakpoint.

Tried `grid-row: 1 / span 2` on Body Kits+Engine and Tires so rows are sized by the shorter items — didn't help enough when Engine still dominated row heights indirectly.

### Why CSS grid can't solve this

There is no pure CSS grid solution that gives independently-sized columns when items in the same grid row have very different heights. This is a fundamental property of how grid rows work — the tallest item in any row defines the height for all columns in that row. No combination of `align-items`, `align-self`, or `grid-row: span` eliminates this.

---

## What works: CSS `columns` with `break-before`

CSS multi-column layout (`columns: 3`) gives each column a truly independent height — no row-height sharing. Items flow top-to-bottom, column-by-column. `break-inside: avoid` on each category keeps it intact. `break-before: column` on specific categories forces a column start at the right points.

**3-col**: `break-before: column` on Drivetrain and Tires
```
Body Kits → col 1
Engine → col 1 (continues)
Drivetrain → forced to col 2
Platform and Handling → col 2 (continues)
Aero and Appearance → col 2 (continues)
Tires and Wheels → forced to col 3
```

**2-col**: Reset Drivetrain/Tires breaks; add `break-before: column` on Platform and Handling
```
Body Kits, Engine, Drivetrain → col 1
Platform and Handling → forced to col 2
Aero, Tires → col 2 (continues)
```

**1-col**: All breaks removed; DOM order takes over.

### Implementation

Six flat `.kit-cat` divs in DOM order. Two CSS modifier classes:

- `up-col3-break` → on Drivetrain, Tires (`break-before: column` at 3-col)
- `up-col2-break` → on Platform and Handling (`break-before: column` at 2-col)

```css
.up-picker { columns: 3; column-gap: 16px; }
.kit-cat { break-inside: avoid; margin-bottom: 16px; }
.up-col3-break { break-before: column; }

@media (max-width: 800px) {
  .up-picker { columns: 2; }
  .up-col3-break { break-before: auto; }
  .up-col2-break { break-before: column; }
}
@media (max-width: 540px) {
  .up-picker { columns: 1; }
  .up-col2-break { break-before: auto; }
}
```

No gaps. No row-height interference. To change which category starts a new column at any breakpoint, just change the class on that category.

---

## Key rule for future layout decisions

**Use CSS grid when items share a common baseline or you want aligned rows across columns. Use CSS `columns` when each column should be independently tall.** The upgrades picker is the latter — column break points are a content decision, not a row-alignment decision.

## Second problem: `break-inside: avoid` fails when a block is taller than the column target height

### Context

The List of Tweaks panel needed a 5-column layout where each section's groups stay together as a block (e.g. all Alignment sub-groups in one column, all Gearing sub-groups in another).

### What failed

**CSS `columns: 5` with `break-inside: avoid`** — Tried this first because it's the established pattern from the upgrades picker. It works for the upgrades list because all blocks (kit-cats) are similar in height. It fails when one block is much taller than the others.

`column-fill: balance` (the default) targets equal column heights. If a block with `break-inside: avoid` is taller than the target column height, browsers break the rule and split the block anyway. Gearing has 7+ rows (Final Drive + 6 gear ratios) while Tires has 2 — Gearing consistently got cut mid-column regardless of `break-inside: avoid`, `min-width: 0`, `display: flex` vs block, or any other CSS tweak.

**CSS grid with `align-items: start`** — Tried this to fix top-alignment (blocks floating mid-column). Reintroduces the row-height sharing problem documented above: Gearing's 7-row height forced an entire grid row to that height, leaving Tires with a huge gap below it.

### What works: explicit column divs + JS height balancing

Render explicit column divs (one per column) and use a greedy JS algorithm to assign tab blocks to the shortest column:

```js
const tweakColumns = computed(() => {
  const cols = Array.from({ length: 5 }, () => [])
  const heights = new Array(5).fill(0)
  for (const entry of changedByTab.value) {
    const [, { groups }] = entry
    let h = 0
    for (const { rows } of groups.values()) h += rows.length + 1
    const shortest = heights.indexOf(Math.min(...heights))
    cols[shortest].push(entry)
    heights[shortest] += h
  }
  return cols
})
```

```css
.ta-nonstock-body { display: flex; gap: 16px; align-items: start; }
.ta-nonstock-col  { flex: 1; display: flex; flex-direction: column; gap: 8px; min-width: 0; }
```

No CSS fragmentation involved. Each column is an independent flex container so blocks can never break. Heights balance naturally because the JS algorithm pre-assigns blocks to the shortest column.

### When to use which approach

- `columns` + `break-inside: avoid`: blocks are similar height, you want the browser to balance automatically.
- Explicit column divs + JS balancing: blocks vary wildly in height (one block may be 3× taller than another), or you need guaranteed no-break behaviour regardless of column height.
