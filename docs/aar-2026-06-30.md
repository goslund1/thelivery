# After Action Report — Session 2026-06-30

## What we were working on

`TuningAdjustments.vue` — restoring arrow key keyboard navigation in the tuning panel.

Expected behavior:
- **Up/Down** — move the focused highlight to the adjacent row
- **Left/Right** — nudge the focused row's slider value by one step

Actual behavior: pressing any arrow key just scrolled the page.

---

## The problem

Two independent bugs, both about focus. The visual "focused" highlight (the magenta border on the right section) is driven by the `focusedKey` reactive ref. **Actual DOM keyboard focus is separate.** The `@keydown="onRowKeydown"` handler on `.ta-row` only fires when a DOM element inside the row has focus. When neither the row div nor its slider had focus, arrow keys fell through to the browser's default scroll behavior.

---

## What we tried that failed

### Attempt 1 — Moving the edit-mode guard

`onRowKeydown` started with `if (!ui.isEditing || r.locked) return` as its very first line, before any `e.preventDefault()` call. When this guard fired (view mode, or locked row), the function returned without calling `preventDefault()`, so the page scrolled.

This was a real issue and was fixed: Up/Down navigation was moved before the guard so it works in all modes, and Left/Right also calls `e.preventDefault()` before checking edit mode.

**But the page still scrolled.** The guard was never the root cause — it only made things worse when something was focused. The real problem was that nothing was focused in the first place.

---

## The root cause: `e.preventDefault()` on mousedown kills focus

`onSliderMouseDown` blocks the slider from jumping to a track click by calling `e.preventDefault()` on the mousedown event:

```js
if (Math.abs(clickPx - thumbPx) > 12) {
  e.preventDefault()
  return
}
```

`e.preventDefault()` on `mousedown` prevents two things:
1. The range input jumping to the clicked position (intentional)
2. **The element receiving keyboard focus (unintentional side effect)**

After this, `onRowClick` fires (click bubbles to `.ta-row`) but bails early because `e.target instanceof HTMLInputElement` (the slider is the target) — so the row div also does **not** get focused.

Result: `focusedKey` shows the visual highlight, but `document.activeElement` is still whatever it was before the click (often `body`). Arrow keys → page scrolls.

This only happens on the **first click** of a slider track. On subsequent clicks, `focusedKey.value === r.key` is already true, so the `if` block is skipped entirely, `e.preventDefault()` is not called, and the slider gets natural focus.

Clicking the thumb (within 12px) never calls `e.preventDefault()`, so the slider always gets natural focus there.

---

## What works: focus the row div manually after preventing default

When blocking the track click, explicitly focus the `.ta-row` parent so keyboard events have a target:

```js
if (Math.abs(clickPx - thumbPx) > 12) {
  e.preventDefault()
  ;(slider.closest('.ta-row') as HTMLElement | null)?.focus({ preventScroll: true })
  return
}
```

The row div has `tabindex="-1"` so it can receive programmatic focus. Once focused, `onRowKeydown` fires on arrow keys: Up/Down navigates to adjacent rows, Left/Right nudges the slider value.

---

## Key rules for future work in this component

**`focusedKey` ≠ keyboard focus.** Setting `focusedKey` makes the row look active. It does not give it keyboard focus. Any code that sets `focusedKey` without also ensuring a DOM element in that row has focus will produce the visual highlight with broken keyboard navigation.

**`e.preventDefault()` on mousedown prevents focus.** Whenever `e.preventDefault()` is called on a `mousedown` event to block a default interaction (jump-to-position, drag-start, etc.), check whether you also need to manually establish focus on something.

**The slider track is the largest click target.** Users will click it first. That's the interaction that needs to work on the first click. Designing around the thumb or the row label as "the real entry point" will always feel broken.
