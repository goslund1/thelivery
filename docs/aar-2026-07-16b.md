## Fixes

### 1. OgMakerModal: content edit now refreshes preview on keystroke
**File:** `frontend/src/components/OgMakerModal.vue`

`@change` fires on blur; `@input` fires on each keystroke. The rotate/shear sliders
were already live via `v-model.number` watchers, but typed content changes weren't
triggering a preview until the user tabbed away. One character change.

---

### 2. Compositor: scrim fires on box overlap, not top-edge position
**File:** `backend/src/compositor.rs`

Old condition: `tb.y > 0.4` ‚Äî checked whether the box's *top* was below 40%.
A tall box starting at y=0.3 and extending to y=0.8 would skip the scrim even though
most of its text was in the lower half of the frame.

New condition: `tb.y + tb.h > 0.45` ‚Äî fires when any part of the box overlaps the
scrim zone. Hasn't caused a visible bug in practice (default spawn is y=0.72) but
would have bitten an unusual layout.

---

### 3. ShareModal: `as any` removed; `recipe` typed via predicate
**File:** `frontend/src/components/ShareModal.vue`

`firstCarName` was using `(r as any).cars?.[0]?.carName` after an `'cars' in r` guard.
The fix: add a type predicate to the `recipe` computed so it returns
`ForzaRecipeSection | null` instead of `Section | null`. All downstream computeds
(`firstCarName`, `shareCode`) get the narrowed type for free; `firstCarName` collapses
to a one-liner.

### 4. `list_og_presets` auth gap closed
**Files:** `backend/src/main.rs`, `frontend/src/components/ShareModal.vue`

`GET /api/og-presets` was the only OG endpoint without `_auth: AuthUser`. Preset data
is non-sensitive today, but the inconsistency was a footgun for future additions.
Added `_auth: AuthUser` to the handler and the `authorization` header to
`fetchPresets()` in ShareModal.

---

### 5. Compositor: ~75 lines of duplication eliminated
**File:** `backend/src/compositor.rs`

Two extraction pairs:

- `render_glyphs(font, text, px, alpha) ‚Üí Option<RgbaImage>` ‚Äî the shared glyph-
  rendering inner loop. `rasterize_text_box` and `rasterize_text_natural` were ~50
  lines each of identical code; they now call `render_glyphs` and differ only in
  whether they Lanczos3-resize the result to a target box.

- `blit_with_transform(canvas, src, ..., pixel_fn)` ‚Äî the shared inverse-mapping
  pixel loop. `blit_transformed` and `blit_transformed_tinted` were ~25 lines each
  of identical setup; they now call `blit_with_transform` with different `pixel_fn`
  closures (`alpha_over` vs tinted blend). All 5 compositor tests still pass.

### 6. Magic style strings replaced with typed constants
**Files:** `backend/src/compositor.rs`, `frontend/src/types.ts`,
`frontend/src/components/OgMakerModal.vue`

`"POSTCARD"`, `"SIGNAL"`, `"GHOST"` appeared as raw strings across both the Rust
compositor and the Vue components. The Rust `font_for_style` match fell through to
`FONT_SIGNAL` for any unrecognized string ‚Äî a typo on the frontend would silently
produce SIGNAL output.

- **Rust:** `TextStyle` enum with `#[serde(rename_all = "SCREAMING_SNAKE_CASE")]` so
  serde deserializes the JSON strings correctly; all match arms and comparisons updated.
- **Frontend:** `OgTextStyle = 'POSTCARD' | 'SIGNAL' | 'GHOST'` union type added to
  `types.ts`; `OgTextBox.style` updated; `STYLES` array and `newStyle` ref in
  `OgMakerModal` typed to `OgTextStyle`.

## Noted but not fixed

**Circular store dependency warning** ‚Äî Vite reports two `[INEFFECTIVE_DYNAMIC_IMPORT]`
warnings for `ui.ts` ‚Üî `modal.ts`. These dynamic `import()` calls exist to break a
circular dependency between the two stores, not to achieve lazy loading. Since both
stores are also statically imported by App.vue and components, the dynamic imports
can't split into a separate chunk (hence the warning). The fix requires restructuring
the stores to extract the shared cross-store logic into a third module ‚Äî meaningful
refactor, deferred.

## Circular store import comments added

`ui.ts` and `modal.ts` are mutually dependent ‚Äî each needs to call into the other
at runtime. The dynamic `import()` workaround was already in place and correct;
added matching explanatory comments to both files (and upgraded the existing
`modal.ts` comment) so the pattern isn't mistaken for lazy-loading or refactored
away. The Vite `[INEFFECTIVE_DYNAMIC_IMPORT]` warning is expected and benign ‚Äî
moving the logic into the component layer to eliminate it would scatter post-login
business logic out of the store.
