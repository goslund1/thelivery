# AAR — 2026-07-17 (c): Live shakedown, wipe-and-reseed, hardening batch

Third session of the day. Started as the plan.md live-site shakedown; the
findings drove a full live wipe-and-reseed, two frontend fixes, and the
pre-launch security batch.

## What was done

1. **Live shakedown** — full pass on https://thelivery.silverleaf.services.
   Healthy: catalog, galleries, lightbox, filters, OG share endpoint (curl),
   deployed bundle byte-identical to local dist. Broken: Light theme
   half-applying; live `/api/cars` empty; card 007 rendered a black void
   (already deleted locally — stale live data); zero-result filters showed
   nothing.

2. **Light theme fix** (`6ef4de5`) — `pickTheme` set `ui.theme` directly,
   which only flips `data-theme`; the nine inline color vars written to
   `<html>` by `theme.load()`/ThemeBuilder override the stylesheet, so the
   page stayed dark while unpinned vars switched. Flyout now routes through
   `themeStore.setAmbiance()` (same path as ThemeBuilder's ambiance row),
   falling back to attribute-only if the store never loaded.

3. **Wipe-and-reseed** (`b0c8e9c`, `486ca03`) — the deploy pipeline only ever
   shipped `cards.json`; `cars.json` and `users.json` never left the repo, so
   live had an empty car registry and no login since first boot. Now: bundle
   ships cards/cars/liveries seeds + users.json injected from the
   `USERS_SEED_JSON` repo secret (gitignored file — password hash); a guarded
   one-time block retired the proof-of-concept live DB to
   `data.db.bak-20260717` (kept on droplet) and the service re-seeded from
   current local data — 10 active cards, Faker + Rally Base Car Test in
   purgatory (kept to exercise admin purge later), 1762 cars, 16 liveries.
   The block was reverted immediately after the deploy (it would have wiped a
   fresh droplet's DB on its second deploy).

4. **Three fresh-DB seeding bugs**, found by production-simulating a fresh
   database and verified with new tests (31 total at that point):
   - Numeric image ids in an exported seed hit `sync_card_images`' UPDATE
     branch — no rows on a fresh DB, so nothing inserts and paths are lost.
     Seed export must strip image ids (insert-by-path branch).
   - Livery-linked images (136/153) silently failed the `livery_id` FK on a
     fresh DB — no liveries seed existed. Failure was invisible because the
     unsynced body kept its paths and the API served them. Added
     `seed_liveries_if_empty` (explicit ids — seeded images reference them)
     running before `normalize_bodies`; sync errors now logged, not swallowed.
   - `normalize_bodies` rewrote migrated cards via `upsert()`, whose
     `ON CONFLICT … deleted_at = NULL` (save = undelete) resurrected
     purgatoried cards at startup. Now a plain body UPDATE. Purgatory travels
     through the seed via a seed-only top-level `deletedAt` marker.

5. **Empty states** (`2b167ed`, `d54e352`) — imageless gallery renders a
   dashed "no images on file" placeholder (16:9 footprint unchanged, controls
   hidden, edit-mode hint); zero-result filter combinations show "no cards
   match" with a Clear Filters button (new `filters.resetFilters()` +
   `hasActiveFilters`; button hidden when no filter is active).

6. **Security batch** (`e6e3d8d`) — plan.md pre-launch items minus the
   deferred CORS/draft-tag pair:
   - `/api/login` rate limit: 5 failed/5 min, 20/hour per anonymized /24,
     DB-backed (`login_rate_log`, migration 0019). Success doesn't count.
   - Path traversal in `trash_image`: `PathBuf::starts_with` is a lexical
     component comparison, so `uploads/../x` passed the old guard. New
     `is_safe_upload_path()` rejects non-plain components at the choke point.
   - `client_ip()` honors X-Forwarded-For — behind Caddy every socket is
     127.0.0.1, so per-address buckets would be one global bucket and five
     bad attempts from anyone would lock out everyone. Applied to login and
     suggestions. Trustworthy because the port is localhost-only.
   - `og:url` absolute (scrapers don't reliably resolve relative).

Backend suite: 29 → 36 tests. All deploys verified live after landing.

## Lessons

- **Verify the table, not the API.** The livery-FK sync failure was invisible
  in `/api/cards` output because failed syncs left paths in the body. The
  "sim passes" call was wrong until the images table itself was checked.
- **A silently swallowed error hid a broken FK for an entire reseed** — the
  `.is_ok()` in normalize's sync step. Log before you skip.
- **Behind a reverse proxy, socket-address rate limiting is a global bucket.**
  End-to-end testing against the deploy topology (not just localhost) is what
  surfaced it.
- **`PathBuf::starts_with` is not a containment check** — it compares
  components lexically and `..` walks right through it.
