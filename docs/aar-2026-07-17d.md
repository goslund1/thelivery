# AAR — 2026-07-17 (d): User roles, audit timeline, temp-password onboarding

Fourth session of the day. Started as "let's take a look at users" with Geoff
as the test case for edit-without-delete permissions; grew into full
role-based access control, an audit timeline, and a forced-password-change
onboarding flow. (Officially these were always on the plan. Wink.)

## What was done

1. **Roles: admin/editor** (`f79ef0c`) — migration 0020 adds `users.role`
   ('admin' default, so existing accounts are untouched). `AuthUser` now reads
   the role from the DB on every request (immediate demotion/revocation, no
   waiting out the 7-day token TTL); new `AdminUser` extractor returns 403 for
   editors. Gating is per-handler via the extractor type.
   - **Editors can**: card CRUD including soft delete (purgatory), image
     upload + trash-delete, livery/tune/car create + update, preset create,
     theme PUT, assess-color (part of the upload flow despite its /admin path).
   - **Admin-only**: all other `/api/admin/*`, hard deletes (livery, tune,
     tuning-preset, og-preset), trash purge/restore, seed export/reload,
     suggestions moderation, and `POST /api/users` — the last one closed a
     real pre-existing hole where any authenticated user could mint accounts.

2. **Audit timeline** (`f79ef0c`) — `audit_log` table + `audit.rs`. Every
   editor-capable mutation records who/what/when. Reversibility guarantees:
   card edits link their `card_history` snapshot version (`detail.prevVersion`);
   card/image deletes reverse via purgatory/trash restore; overwrites with no
   other history (livery/tune/theme/og-preset updates) store the previous
   state as JSON in `detail.prev`. Admin Panel gained a **Timeline** tab
   (newest-first, paged via `before_id`, color-coded by action kind).
   Recording is best-effort — an audit insert failure never fails the request.

3. **Temp-password onboarding** (`7a42481`) — migration 0021 adds
   `users.must_change_password`. While set, the auth extractor 403s every
   request except `PUT /api/me/password`, so enforcement is server-side.
   Add User form gained a role select (default editor) and a "temporary
   password" checkbox (default on). `ForcePasswordModal` is mounted last in
   App.vue and deliberately NOT registered with the modal store — no Escape,
   no click-outside, `closeTopModal` can't reach it; "Sign out instead" is the
   only other exit and keeps the temp password valid for a later retry. Flag
   persists in localStorage so a refresh can't dodge the modal.

## Gotchas / lessons

- **Existing sessions must re-login after the roles deploy** — pre-role tokens
  work (the backend reads role from the DB) but localStorage has no
  `auth_role`, so the UI hides admin features until a fresh login.
- `adduser` CLI needs a tty for the password prompt; in automation, piping
  stdin fails (`Device not configured`) — use `expect`.
- Suggestions badge/viewer moved behind the admin gate because dismissing a
  suggestion is a hard delete. Revisit if Geoff should triage.
