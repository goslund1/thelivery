**Evidence note:** The race condition in finding #1 was reproduced live in the browser with the

## Correctness — RecipeSection / CarTabs flow

### Race condition in pending preset auto-apply
**File:** `frontend/src/components/RecipeSection.vue` (watch on `activeVariantIndex`)

The watcher is `async` and does two awaits: `api.listTuningPresets()` then `nextTick()`. If the
user switched tabs during either await, `taRef.value.applyPresetValues()` fired against the new
tab's TuningAdjustments widget and `flush()` wrote the result into the wrong variant slot.

**Fix:** two `if (activeVariantIndex.value !== idx) return` guards, one after each await.

---

### Reset token watch didn't reset activeVariantIndex
**File:** `frontend/src/components/RecipeSection.vue` (resetToken watcher)

On history restore or discard, `Object.assign(local, cloneRecipe(props.recipe))` replaced the
variant array but `activeVariantIndex` was not reset. If the restored snapshot had fewer variants
than the current index, the tab strip rendered with no tab appearing active in edit mode.

**Fix:** `activeVariantIndex.value = 0` at the top of the watcher before the assign.

---

### flush() left local.variants[activeIdx].adjustments one step stale
**File:** `frontend/src/components/RecipeSection.vue` (`flush()`)

`v.adjustments = local.adjustments` was written before the clone, but the correct live value from
`taRef.getAdjustments()` only went into the outgoing clone — not back into `local.variants`. The
emitted clone was always correct and the tab-switch snapshot read from `taRef` directly, so no
visible bug existed, but the in-memory invariant was violated and could bite future changes.

**Fix:** added `local.variants![activeVariantIndex.value].adjustments = liveAdj` after the taRef
block in `flush()`.

## Architecture — API error handling

### Unsafe `(e as Error).message` cast (8 sites)
The dominant pattern post-session-12 is `e instanceof Error ? e.message : String(e)`. Eight older
files used the unsafe cast, which returns `undefined` if the thrown value isn't an Error instance.

**Sites fixed:**
- `stores/cards.ts`
- `stores/theme.ts` (×2)
- `components/SuggestionViewer.vue` (×2)
- `components/EditCardModal.vue`
- `components/CardHistoryModal.vue`
- `components/NewCardModal.vue` (×2)

**Fix:** extracted `errMsg(e: unknown): string` to `utils/errMsg.ts` and replaced all sites.

---

### Duplicate local errMsg() definitions (2 files)
`AdminPanel.vue` and `UserSettingsModal.vue` each defined an identical local helper.
`ImageMigrationModal.vue` and `TuningAdjustments.vue` inlined the same logic.

**Fix:** all four now import from `utils/errMsg.ts`.

---

### Outlier fallback in TuningAdjustments
`TuningAdjustments.vue` used `'Something went wrong.'` as the non-Error fallback instead of
`String(e)`, silently discarding the actual error text.

**Fix:** replaced with `errMsg(e)`.

## Architecture — Modal state management

### ThemeBuilder open state outside modal store
Every other panel's open state lives in `modal.ts`, enabling `closeTopModal()` to close them in
z-order on Escape. `themeBuilderOpen` was a local `ref(false)` in `SideBug.vue`, so Escape had
no effect on the ThemeBuilder.

**Fix:** moved `themeBuilderOpen / openThemeBuilder / closeThemeBuilder` into `modal.ts`, wired
into `closeTopModal()` at the same priority level as the admin panel, updated `SideBug.vue` to
read from the store.

## Type-safety

### null as unknown as string in tuning.ts
`SlotMapping.subcategory` was typed `string`, but the `arb` and `aero` entries set it to `null`
(masked with `null as unknown as string`) because those tabs are handled by special-case branches
that `continue` before the generic subcategory path runs.

The type system was not protecting anything — a future mapping with null subcategory would silently
pass null to `hasSubcategory()` as a string.

**Fix:** changed `subcategory: string` to `subcategory: string | null` in the interface, removed
the casts, added a `mapping.subcategory &&` guard at the one generic call site. Runtime behavior
unchanged.

## Race condition repro — demonstrated 2026-07-12

**Setup:** Smokin (card "1") had `pendingPresetId: 2` on the Corvette variant and `pendingPresetId: 3`
on Sprite MkI from a previous wizard run. A `fetch` monkey-patch delayed the `tuning-presets`
endpoint by 3 seconds.

**Sequence:**
1. Clicked the Chevrolet Corvette #777 tab — watcher fired, fetch started (3s delay confirmed in console).
2. After 1 second (well within the 3s window), clicked back to the 599D GTB Fiorano #117 anchor tab.
3. After 5 seconds (2s after the fetch completed), observed anchor slider values and pendingPresetIds.

**Result:**
- Anchor sliders before: `[32, 31, 26, 18, 0, 0, 0, 0]`
- Anchor sliders after: `[32, 31, 26, 18, 0, 0, 0, 0]` — unchanged; anchor not corrupted.
- `pendingPresetIds` after: `[null, 2, 3]` — Corvette's `pendingPresetId: 2` still set, meaning the
  guard returned early before `delete v.pendingPresetId`. The pending apply is deferred, not lost —
  the next click on the Corvette tab will re-trigger the load.

**Verdict:** Guard confirmed effective. The race was triggerable (fetch fired, tab switched mid-flight)
and the guard prevented the preset from being written to the wrong variant. Finding upgraded from
inferred to **demonstrated**.
