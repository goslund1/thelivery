## Transmission tier detection ‚Äî three layers of the same bug

### What we wanted
Gear sliders in the Gearing tab should unlock when a card has a Race or Drift transmission installed. Final Drive unlocks for Sport+.

### What kept going wrong

**Layer 1 ‚Äî Wrong source of truth.** `transmissionTier` was reading from `cards.byId(props.cardId)` ‚Äî the Pinia store's copy of the card. But `RecipeSection` passes its own `local` reactive clone of the recipe to `TuningAdjustments`. The store copy and `local` are separate objects. Any upgrades edited in the current session existed only in `local`; the store hadn't been updated yet. Result: `transmissionTier` always saw the pre-edit state.

Fix: pass `upgrades` and `coreSpecs` as props from `RecipeSection` down to `TuningAdjustments`, and read from those props instead of the store.

**Layer 2 ‚Äî String matching too narrow.** The upgrade check was `part.startsWith('Race') && part.includes('Transmission')`. Real card data has entries like "9-Speed Racing Transmission" and "10-Speed Racing Transmission" ‚Äî neither starts with "Race". The coreSpec check `spec.includes('Race')` was theoretically correct ("Racing" contains "Race" as a substring) but was masked by Layer 1.

Fix: broadened to `part.includes('Transmission') && (part.includes('Race') || part.includes('Racing'))`.

**Layer 3 ‚Äî Legend card had placeholder data.** Card 000 is the reference example used to verify all UI patterns. Its coreSpec was `"e.g. stock"` ‚Äî intentionally wrong placeholder copy ‚Äî so it always computed tier `'none'` and locked all sliders. Fixed by updating the legend card's Transmission coreSpec to `"9-Speed Race"` directly in SQLite and re-exporting the seed.

### Key rule
When a Vue component receives props that are a deep-cloned reactive object from its parent, **never bypass those props to fetch the same data from a store**. The clone and the store are separate reactive graphs. Edits to the clone are invisible to the store until saved.

## `hasSuggestionData` temporal dead zone crash

### What happened
All cards stopped rendering after a hard refresh. The page showed only the header.

### Root cause
In `TuningAdjustments.vue`, `watchEffect` was declared before `hasSuggestionData`:

```ts
const showSuggestBar = computed(() => !ui.isEditing && hasSuggestionData.value && ...)
watchEffect(() => { if (showSuggestBar.value) ... })  // line ~805
// ...
const hasSuggestionData = computed(...)  // line ~818
```

`watchEffect` runs immediately on setup. Its callback reads `showSuggestBar`, which reads `hasSuggestionData` ‚Äî which hasn't been declared yet (temporal dead zone in `<script setup>`). This threw a `ReferenceError` that silently killed every `TuningAdjustments` instance, and since every card has one, no cards rendered.

Fix: moved `hasSuggestionData` above `showSuggestBar` and `watchEffect`.

### Key rule
In `<script setup>`, computed refs and reactive state must be declared before any code that reads them ‚Äî including `watchEffect`, `watch`, and other computeds. Unlike `var` hoisting, `const`/`let` declarations in setup are not hoisted. A `watchEffect` that runs immediately will throw if it touches an undeclared ref.

## Vite HMR gotcha ‚Äî import added mid-file

When `useModalStore` was imported inside `EditCardModal.vue` after the `defineProps` call (due to an edit that inserted an import in the wrong position), Vite's HMR silently stopped recompiling that component. The browser continued serving the previous cached version. Hard refreshes had no effect because the server itself was handing out the stale transform.

Fix: moved the import to the top of the `<script setup>` block where all imports belong. Restarting the Vite dev server was required to flush the stale transform.

### Key rule
All imports in `<script setup>` must appear at the top, before `defineProps`/`defineEmits`. An import placed after these compiler macros will cause Vite HMR to silently break for that file. The symptom is: code changes on disk have no effect in the browser even after hard refresh. The fix is always: move imports to the top, then restart the dev server.

## Card version history ‚Äî what was built

`CardHistoryModal.vue` ‚Äî a full-screen overlay (triggered from the History button in EditCardModal's top-right corner) that:
- Lists all saved versions from `/api/cards/:id/history`, newest first
- Loads the full card body for any selected version
- Shows a structured diff against the current card state: slider values changed, upgrades added/removed, core specs changed, text section changes, top-level field changes
- Restore button replaces the card's in-memory state with the historical version and marks it dirty ‚Äî the user controls when it actually saves

Supporting changes:
- `modal.ts`: `historyCardId`, `openHistory`, `closeHistory`
- `cards.ts`: `restoreCardVersion` ‚Äî overwrites mutable fields in-place, preserves id/catalogNumber/isLegend

## Debugging lesson: confirm which URL you're looking at

A significant amount of time was spent trying to fix a "missing button" that was likely already working. The cause: Safari had opened new tabs to the live production site, and the session reverted to viewing that instead of localhost. Hard refreshes on the live site had no effect on local code changes.

Before debugging a UI issue, confirm `window.location.href` in the console or check the address bar. localhost:5173 and thelivery.silverleaf.services look nearly identical.
