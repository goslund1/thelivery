## Item 1 — ThemeBuilder effects sliders with color swatches

- An opacity slider
- A color swatch button that wires into the ColorPicker array

## Bug — tire pressure sliders triggering transmission dialog

Tire pressure rows were being locked by the same re-lock logic that's intended only for gearing rows (`if (!autoAddedPart.value && Math.abs(r.value - r.stock) <= r.step / 2) r.locked = true`). Once locked, any slider interaction called `unlockByUpgrade()` which opened the transmission picker.

Fix: gated both the `unlockByUpgrade()` call and the re-lock line on `r.tab === 'gearing'`. For non-gearing locked rows, the locked flag is now silently cleared on first interaction (`else r.locked = false`) rather than blocking or opening a dialog.

---

## Bug — arrow key navigation leaving component scope

The Up/Down row navigation used `document.querySelector('.ta-slider[data-key="..."]')` — document-wide. With multiple TuningAdjustments instances on the page (one per card in stacked view), this could focus a slider in a different card entirely, causing that card's locked gearing dialog to open unexpectedly. This was the "old style dialog then new style dialog" report.

Fix: scoped all querySelector calls to `taRef.value` (the component's own root element). Also converted the single-step lookup to a loop that skips rows whose sliders aren't in the DOM — handles tab mode correctly, where only the active tab's rows are rendered and adjacent-tab rows return null.

The same scope fix was applied to the post-confirm focus call in `onTransChoice`.

---

## Self-inflicted regression — buildLocalRows reorder

In an attempt to make tires directly reachable from alignment without stepping through gearing rows, `buildLocalRows` was modified to always append gearing at the end. This broke gearing keyboard navigation because the localRows order no longer matched the visual section order (CANONICAL_TABS puts gearing second, between tires and alignment). Navigation into the gearing section from alignment became impossible.

Reverted immediately. The correct fix was already in place: the navigation loop skips invisible rows, which handles the actual underlying issue (tab-mode rows not in DOM). No reordering needed.

**Lesson:** don't restructure core data ordering to paper over a navigation edge case. The loop already solved it.

## Item 2 — AI color assessment endpoint

`POST /api/admin/liveries/:id/assess-color` (Axum, auth-gated). Calls Claude claude-haiku-4-5-20251001 with the livery's thumbnail (thumb_path preferred — 30-50x fewer tokens than full-res; falls back to path). Frontend: `api.assessLiveryColor(id)` in `api.ts`. PhotoDetail.vue got an inline assess log: tagging a livery auto-triggers assess, shows "assessing…", updates to "Gold / Black" on completion, fades after 2s.

## Item 3 — Batch import flow in NewCardModal

Replaced the old one-shot import path with a full batch pipeline:

1. User stages photos → photo setup row appears (CarPicker + livery name input)
2. Livery name defaults to "Livery Name"; must be edited to enable Import button
3. Click Import → create card, create livery, then launch **all uploads in parallel** (XHR-based `uploadImageWithProgress` with `liveryId` field so each image row gets `livery_id` immediately)
4. Progress log populates with all filenames at once; each row's background is a CSS `linear-gradient` progress bar driven by a `--prog` custom property updated per XHR `onprogress` event
5. After first upload resolves, `assessLiveryColor()` fires; assess row appears at bottom of log showing "assessing… → Gold / Black"
6. When assess settles: 2s hold → log fades out → modal closes

The per-photo progress bar approach is pure CSS — no canvas, no animation frames. `--prog` is updated directly on the element style; the linear-gradient repaints on every XHR progress tick.

## Item 4 — Image identity architectural refactor

**Problem:** Card body JSON stored image paths as strings — paths were de-facto identifiers. The `images` table existed but was secondary; card body was the source of truth. Path changes would break references; reusing an image across cards required file duplication.

**Fix:** `images` table is now the single source of truth.

**Backend (main.rs):**
- `fetch_images_for_card()` — SELECT all images for a card, return as JSON array
- `inject_images()` — on every card read, replace `body["images"]` with DB rows (full data: path, thumbPath, stagePath, livery_id, car_id, alt, sort_order)
- `sync_card_images()` — on every card write, upsert `images` table from body (numeric id → UPDATE existing row; path-only string → find-or-INSERT); strips paths from body before save
- `normalize_bodies()` step 3 — at startup, migrates any cards whose body still contains path-string images: calls `sync_card_images`, saves stripped body. Idempotent.
- `list_cards`, `get_card` → call `inject_images` after parse
- `put_card`, `create_card` → call `sync_card_images` before upsert, `inject_images` after

**Frontend:**
- `CardImage.id: number` (was `string`) — always the DB integer PK
- Negative temp ids (`--_imageIdCounter`, starts at 0, goes negative) used only in `addImageToPool` for the instant before a server round-trip returns the real id — never collides with positive DB auto-increment ids
- All `imageId: string` params → `number` across `cards.ts`, `Gallery.vue`, `ImagePicker.vue`, `PhotoDetail.vue`
- `Set<string>` → `Set<number>` in `selectedIds` and `pendingDeleteId`

One TS error caught during build: `sortedImages.value?.length` — `sortedImages` was a plain array, not a ref. Fixed immediately.

## Item 5 — ImageMigrationModal.vue

Admin tool (SideBug → Filters flyout → "Image Migration" button, auth-gated) for retrofitting existing cards into the new image identity system.

Walks every card that has images, one at a time:
- **Header:** counter (n / total) + card name
- **Thumbnail strip:** scrollable horizontal strip of all card images (thumb preferred)
- **Setup:** CarPicker + livery name input (same validation as NewCardModal — must differ from default)
- **Process card →:** creates livery, calls `setImageMeta` on all images (sets `livery_id`, `carId`), saves card (syncs images table), fires `assessLiveryColor`. Progress log with sweeping accent-tinted bg bar, assess row appended, auto-fades + advances to next card
- **Skip →:** skips without processing (for cards that don't need livery tagging)
- **Done state:** "All N cards processed" + close button

Per-card state resets via a `watch(currentCard)` — carId seeds from `card.carId`, livery name seeds from `card.name`. Reopen resets index to 0.

Modal wired into `modal.ts` (`imageMigrationOpen`, `openImageMigration`, `closeImageMigration`) and mounted in `App.vue`.
